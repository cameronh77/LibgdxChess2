JSON Refactor — Change Summary

This document records every file that was created, modified, or deleted during the zone-data refactor. The goal was to merge three separate JSON files into one, eliminate parallel-array fragility, and switch piece layouts from compact single-character encoding to readable full-name strings.

Why it was done

Old problemNew approach
Three separate files (mapData.json, enemyData.json, shopData.json) all keyed on the same map name — must stay in sync manually.One file: zoneData.json — everything for a zone in one place.
Parallel arrays (mapLayouts[], locationInternals[], startingPositions[]) linked only by array index — easy to drift out of sync.Bundled variants[] objects — each variant carries its own player start and node list.
Compact piece encoding like "gq33" — goblin queen at (3,3). Runs out of letters, unreadable.Structured objects: { "piece": "goblin-queen", "x": 3, "y": 3 }.
Separate randomEnemies / randomShops lists duplicated data already in the main enemy/shop arrays.Single fixed: boolean on each enemy/shop entry.

Files created (JSON)

assets/jsons/zoneData.json NEW

Top-level array of zone objects. Each zone has:

{
  "mapName": "classic4",   // used as the lookup key
  "mapX": 4, "mapY": 4,   // board dimensions
  "variants": [ ... ],    // random layout selection pool
  "enemies":  [ ... ],    // enemy definitions referenced from variants
  "shops":    [ ... ]     // shop definitions referenced from variants
}

Variant structure — one variant is picked at random each time a map is entered:

"variants": [
  {
    "playerStart": { "x": 0, "y": 1 },
    "nodes": [
      { "type": "enemy", "x": 0, "y": 0, "ref": "random" },
      { "type": "shop",  "x": 3, "y": 3, "ref": "shop1"  }
    ]
  }
]

Enemy definition — fixed: true means referenced only by ID; false means eligible for "random" pool:

"enemies": [
  {
    "enemyId": "classic1",
    "fixed": false,
    "enemyLayout": [
      { "piece": "goblin-queen", "x": 3, "y": 3 }
    ],
    "dialogue": null,
    "rewards": {
      "portals": ["classic6", "classic8"],
      "portalLocations": [{ "x": 0, "y": 3 }, { "x": 1, "y": 3 }],
      "currency": 3
    }
  }
]

engine/JsonLoader.java MODIFIED

Replaced three separate load methods (loadMapData, loadEnemyData, loadShopData) with one:

public void loadZoneData() {
    // Explicit element-type hints for libGDX's JSON deserialiser:
    json.setElementType(ZoneData.class,    "variants",        ZoneVariant.class);
    json.setElementType(ZoneData.class,    "enemies",         EnemyData.class);
    json.setElementType(ZoneData.class,    "shops",           ShopData.class);
    json.setElementType(ZoneVariant.class, "nodes",           MapNode.class);
    json.setElementType(EnemyData.class,   "enemyLayout",     PiecePlacement.class);
    json.setElementType(Rewards.class,     "portals",         String.class);
    json.setElementType(Rewards.class,     "portalLocations", Coordinates.class);
    json.setElementType(ShopData.class,    "shopLayout",      IndividualWare.class);

    ArrayList<ZoneData> zoneList = json.fromJson(ArrayList.class, ZoneData.class, ...);
    for (ZoneData zone : zoneList) zones.put(zone.mapName, zone);
}

Also added getEnemyData(String map, String ref) and getShopData(String map, String ref) — both support ref="random" (picks a non-fixed entry at random) and a named ID lookup.

location/board/MapBoard.java MODIFIED

Constructor now accepts ZoneVariant variant instead of raw layout strings. populateBoard() iterates variant.nodes and switches on node.type ("enemy" / "shop") to populate the map.

Outstanding issue

Enemy piece positions default to (0,0) in the match board.
The setElementType calls were added to fix a suspected libGDX JSON generic-type inference failure for ArrayList<PiecePlacement>, but the problem persists. Next steps:
  • Add System.out.println in BotPlayer.createPieceList() to print placement.piece, placement.x, placement.y — confirms whether the JSON data is reaching the code correctly.
  • Add a print in MapBoard.populateBoard() to print node.x, node.y — checks if map node positions are also 0 (would indicate a wider int-field deserialization failure).
  • If both are 0: consider switching int x, int y fields to float x, float y in PiecePlacement and MapNode (libGDX JSON stores numbers as floats internally; conversion to int may silently fail in some versions).
  • If only PiecePlacement is 0: try wrapping the position in a Coordinates object (same pattern as IndividualWare.location, which is known to work).
