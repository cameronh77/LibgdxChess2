UI Refactor — Panel System

Explains the changes made to how the game screen renders its three UI panels.

Why This Was Changed

The original GameScreen managed all three side panels and the bottom bar in one class. As more things were added (king powers, pre-game buttons, passive display), it started accumulating a growing list of individual setVisible() calls scattered through render(): 

// OLD render() — more conditions every time something was added
if (state == CHANGING_MAP)       { /* resize board */ }
if (state == CHANGING_INVENTORY) { rebuildInventory(); }
if (state == CHANGING_POWERS)    { rebuildActivePowers(); }
alterLayoutBtn.setVisible(!(board instanceof MatchBoard));
preGameTable.setVisible(board instanceof AlterLayoutBoard);
inventoryTable.setVisible(board instanceof AlterLayoutBoard);

Each new UI element would need its own condition. This approach doesn't scale.

The New Idea: Panels Decide Their Own Content

Each panel is now its own class. When the board state changes, GameScreen tells every panel "the board is now X" and each panel decides what to show by itself.

┌─────────────────────────────────────────────┐ │ GameScreen (coordinator) │ │ │ │ "Board changed!" → leftPanel.refresh(board)│ │ → rightPanel.refresh(board)│ │ → bottomPanel.refresh(board)│ └─────────────────────────────────────────────┘ │ │ │ LeftPanel RightPanel BottomPanel decides decides decides what to show what to show what to show

GameScreen.render() is now just 6 lines — it checks for a state change, calls refresh() on each panel, and resets the state back to neutral.

GameState Simplification

Previously there were three separate states to signal different kinds of UI updates:

Before

NEUTRAL
CHANGING_MAP        // board size changed
CHANGING_INVENTORY  // items changed
CHANGING_POWERS     // powers changed

After

NEUTRAL
BOARD_STATE_CHANGED // anything changed

All three old states caused the same response — rebuild the UI. So they've been collapsed into one. Anything that used to set CHANGING_MAP, CHANGING_INVENTORY, or CHANGING_POWERS now sets BOARD_STATE_CHANGED.

The Three Panel Classes

Each panel extends libGDX's Table and has a single refresh(Board board) method.

LeftPanel — the action sidebar

Board typeWhat it shows
MatchBoard (battle)Nothing — completely empty
MapBoard / ShopBoardThe "change layout" button
AlterLayoutBoard (placement)"Change layout" button + any pre-game king power buttons

RightPanel — the status sidebar

Board typeWhat it shows
All boardsGold counter + passive king power list
AlterLayoutBoard (placement)+ piece inventory (drag to place)

RightPanel also has an updateCurrency() method that GameScreen calls every frame to keep the gold display live — currency can change at any moment, so it can't wait for a state change.

BottomPanel — king active powers

Board typeWhat it shows
MatchBoard (battle)Up to 3 active king power buttons (greyed out if unavailable)
All other boardsNothing

The New render() Loop

public void render(float delta) {
    ScreenUtils.clear(Color.BLACK);

    if (gameRunInstance.getGameState() != GameState.NEUTRAL) {
        Board current = gameRunInstance.getCurrentBoard();
        boardActor.setSize(board.getPixelWidth(), board.getPixelHeight());
        centerContainer.invalidateHierarchy();
        leftPanel.refresh(current);    // left panel decides what it shows
        rightPanel.refresh(current);   // right panel decides what it shows
        bottomPanel.refresh(current);  // bottom panel decides what it shows
        gameRunInstance.setGameState(GameState.NEUTRAL);
    }

    rightPanel.updateCurrency();  // live gold counter, updates every frame

    uiStage.act(delta);
    uiStage.draw();
}

Adding a new UI element to a panel in future no longer requires touching GameScreen at all — you just modify the relevant panel's refresh() method.
